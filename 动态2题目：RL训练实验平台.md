# 题目：构建 RL 训练实验平台——以 2048 为训练场景

---

## 一、背景

### 现状

你入职一家 AI 公司。公司正在为客户搭建一套强化学习训练平台（多智能体协同控制方向），客户要求先看到一个**可运行的 Demo**来验证技术可行性。

技术负责人选定 **2048 游戏**作为 MVP 训练场景，原因：

- 纯 Python 实现，零外部仿真器依赖
- 训练 30 分钟 ~ 2 小时即可看到收敛
- 策略有深度（贪心策略只能到 512，好策略能到 2048+），区分度好
- 结果直观——最大方块从 256 涨到 1024+，一眼可见 Agent 在变聪明

**你是这个项目的唯一工程师**，需要独立完成从需求理解到系统上线的全部工作。

### 你接到的需求

技术负责人给你发了以下原始需求：

> 用 2048 游戏作为训练场景，搭建一个 RL 训练实验平台的 MVP。
>
> 1. 实现一个 2048 游戏环境，符合 Gymnasium（`reset` / `step` / `render`）标准接口
> 2. 实现至少一种 RL 算法（DQN 或 PPO），超参数通过配置文件管理
> 3. 训练过程中自动记录：实验配置、训练指标（reward 曲线）、模型检查点
> 4. 提供一个可视化界面，能查看训练曲线、对比不同实验
> 5. 系统可一键启动（Docker 或脚本）
> 6. 有测试，能证明系统是正确的
>
> 2048 游戏规则自行查阅。环境需从零实现。
>
> 这就是全部输入。怎么理解、怎么拆、怎么做，你决定。

## 二、研发流程：G0 → G4 门控体系

公司采用「AI 执行、人判断」的研发模式。你需要用 AI 工具完成每个阶段的执行，同时在关键节点做出人工判断。整个流程分为五个门控节点（你可以进行优化或者修改）：

```
原始需求
  │
  ▼
[G0] 需求结构化  ── 你用 AI 把模糊需求转化为精确 Spec，识别并解决歧义
  │
  ▼
[G1] 拆解 + 测试库  ── 你用 AI 把 Spec 拆成 Epic/Story，生成测试用例；你补充 AI 遗漏的场景
  │
  ▼
[G2] 任务规划  ── 你用 AI 把 Story 拆成具体 Task，确定技术方案，标注风险
  │
  ▼
[G3] 开发 + Review  ── 你用 AI 写代码，然后对 AI 的代码做 Code Review，输出结构化报告
  │
  ▼
[G4] 验收  ── 你运行测试库，确认功能满足 G1 锁定的验收标准
  │
  ▼
交付完成
```
